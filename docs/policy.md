# Policy Document — Stage Portfolio Delivery

> Updated: 2026-04-22 | Status: active draft

---

## 1. Goal

이 문서는 스테이지형 포트폴리오 프로젝트를
어떤 단위의 이슈로 나누고, 어떤 순서로 구현하며,
각 이슈가 무엇을 완료 상태로 보는지 정의한다.

핵심 목적은 아래와 같다.

- 이슈가 현재 기획과 정확히 연결될 것
- 한 이슈가 하나의 관심사만 다룰 것
- PR 단위가 과도하게 커지지 않을 것
- 구현 순서가 실제 의존관계와 맞을 것

---

## 2. Issue Design Principles

### Principle 1. Feature Slice First

이슈는 파일 기준이 아니라 사용자 경험 기준으로 자른다.
즉, "컴포넌트 3개 수정"이 아니라
"시작 화면 구성"처럼 사용자가 인지하는 기능 단위로 만든다.

### Principle 2. One Clear Outcome

하나의 이슈는 완료 여부를 명확히 판단할 수 있어야 한다.
완료 조건이 여러 갈래로 퍼지면 이슈를 분리한다.

### Principle 3. Respect Dependency Order

구조를 만드는 이슈가 먼저,
그 위에 콘텐츠와 개별 스테이지 이슈가 올라가야 한다.
즉, 진행도 store 없이 스테이지 구현부터 시작하지 않는다.

### Principle 4. Portfolio Before Game Difficulty

게임성보다 정보 전달이 우선이다.
이슈도 액션 난이도보다 다음 항목을 우선 순위로 둔다.

- 진행도 노출
- 이력서 조각 해금
- 최종 Resume View
- 읽기 쉬운 콘텐츠 구조

### Principle 5. Reusable Game Patterns

미니게임은 스테이지마다 전부 새로 만들지 않는다.
재사용 가능한 패턴을 2~3개 정하고,
콘텐츠와 연출만 바꾸는 방향을 우선한다.

---

## 3. Delivery Order

이 프로젝트의 권장 구현 순서는 아래와 같다.

1. 시작 화면과 스테이지 선택 구조를 만든다.
2. 진행도 상태와 Resume Blueprint를 만든다.
3. 결과 화면과 해금 플로우를 만든다.
4. 개별 스테이지를 순차적으로 붙인다.
5. 최종 Resume View를 만든다.
6. 마지막에 접근성과 반응형을 정리한다.

이 순서를 벗어나면,
플레이는 가능해도 포트폴리오로서의 완성도가 늦게 붙는 문제가 생긴다.

---

## 4. Issue Buckets

새 이슈는 아래 4개 버킷으로 관리한다.

### Bucket A. Core Structure

- 시작 화면
- 스테이지 선택 화면
- 라우팅 또는 뷰 전환
- 공통 레이아웃

### Bucket B. Progress System

- 진행도 store
- Resume Blueprint UI
- 스테이지 해금 상태
- 결과 반영 로직

### Bucket C. Stage Content

- 회사별 경력 스테이지
- 학력/대외활동 스테이지
- 경력 상세 스테이지
- 프로젝트/최종 해금 스테이지

### Bucket D. Finalization

- 최종 Resume View
- 반응형
- 접근성
- 폴리시/텍스트/QA 정리

---

## 5. Definition of Done

각 이슈는 아래 조건을 만족해야 완료로 본다.

### Common Done Criteria

- 화면 또는 기능이 실제로 동작한다.
- 기획 문서와 충돌하지 않는다.
- 현재 진행률 또는 해금 상태와 연결된다.
- 모바일 퍼스트 기준을 해치지 않는다.
- 접근성 기본 조건을 크게 어기지 않는다.

### Stage Issue Done Criteria

- 스테이지 진입이 가능하다.
- 플레이 규칙이 짧게 이해된다.
- 클리어 조건이 명확하다.
- 클리어 시 Resume Fragment가 열린다.
- 결과 화면 또는 진행률 변화가 확인된다.

### Resume Issue Done Criteria

- 해금된 정보만 노출된다.
- 섹션 구분이 분명하다.
- 채용 담당자가 스캔 가능한 밀도로 정리된다.

## 6. Recommended Issue Set

현재 기획 기준으로 새 이슈는 아래 구성이 가장 적절하다.

### Issue 1. 시작 화면 및 스테이지 선택 구조 구현

- 범위:
  - Start Screen
  - Stage Select / World Map
  - 기본 뷰 전환 구조
- 결과:
  - 사용자가 게임을 시작하고 다음 스테이지를 고를 수 있다.

### Issue 2. 진행도 상태 관리 및 Resume Blueprint 구현

- 범위:
  - 진행도 store
  - 클리어/해금 상태 모델
  - Resume Blueprint UI
- 결과:
  - 사용자가 현재 이력서 완성도를 항상 확인할 수 있다.

### Issue 3. 스테이지 결과 화면 및 해금 플로우 구현

- 범위:
  - 클리어 결과 화면
  - Resume Fragment 보상 표시
  - 다음 스테이지 해금 처리
- 결과:
  - 스테이지 완료 후 보상과 다음 행동이 명확하게 연결된다.

### Issue 4. 경력 스테이지 구현 - Toss

- 범위:
  - 토스 경력용 미니게임
  - 토스 이력서 조각 콘텐츠
- 결과:
  - 토스 관련 경력 정보를 스테이지 클리어 보상으로 확인할 수 있다.

### Issue 5. 경력 스테이지 구현 - Lighthouse

- 범위:
  - 라이트하우스 경력용 미니게임
  - 관련 이력서 조각 콘텐츠
- 결과:
  - 라이트하우스 경력 정보를 보상으로 확인할 수 있다.

### Issue 6. 경력 스테이지 구현 - Rowlabs

- 범위:
  - 로랩스 경력용 미니게임
  - 관련 이력서 조각 콘텐츠
- 결과:
  - 로랩스 경력 정보를 보상으로 확인할 수 있다.

### Issue 7. 학력 및 대외활동 스테이지 구현

- 범위:
  - Education & Activities 스테이지
  - 관련 이력서 조각 콘텐츠
- 결과:
  - 학력과 대외활동 정보를 하나의 성장 서사로 확인할 수 있다.

### Issue 8. 경력 상세 역량 스테이지 구현

- 범위:
  - 협업 방식
  - 문제 해결 방식
  - 강점 키워드
- 결과:
  - 일하는 방식과 강점을 별도 스테이지로 전달할 수 있다.

### Issue 9. 프로젝트 하이라이트 및 최종 해금 스테이지 구현

- 범위:
  - 프로젝트 하이라이트 콘텐츠
  - 마지막 스테이지
  - 최종 Resume Open 조건 연결
- 결과:
  - 모든 스테이지를 완료하면 최종 이력서가 열린다.

### Issue 10. 최종 Resume View 및 연락 수단 구현

- 범위:
  - 완성된 이력서 전체본
  - 프로젝트 요약
  - GitHub / 이메일 / 링크
- 결과:
  - 게임 완료 후 실제 제출 가능한 포트폴리오 요약 화면이 제공된다.

### Issue 11. 반응형 및 접근성 정리

- 범위:
  - 모바일 레이아웃
  - 키보드 조작
  - 대비 / 포커스 / 모션 감소 대응
- 결과:
  - 주요 화면이 모바일과 접근성 기준에서 최소 품질을 확보한다.

## 7. Label Policy

새 이슈는 아래 라벨 기준을 우선 사용한다.

- `enhancement`: 기능 구현 이슈
- `refactor`: 구조 개선 이슈
- `question`: 결정이 필요한 이슈

현 시점 기준으로 새 이슈 대부분은 `enhancement`가 적절하다.

---

## 8. Branch / PR Policy

- 브랜치명은 이슈 단위로 만든다.
- PR은 하나의 이슈만 담는다.
- 결과 화면 + 진행도 + 스테이지 구현처럼 관심사가 다르면 분리한다.

권장 예시:

- `feature/issue-start-stage-select`
- `feature/issue-progress-blueprint`
- `feature/issue-stage-toss`

---

## 9. Anti-Patterns

아래 방식은 피한다.

- 3개 이상의 스테이지를 한 이슈에 묶기
- 진행도 시스템 없이 스테이지부터 여러 개 만들기
- 게임 UI와 최종 Resume View를 한 PR에 모두 넣기
- 퍼즐 난이도를 높이는 데 시간을 쓰고 포트폴리오 정보 연결을 미루기

---

## 10. Operating Rule

기존 기획과 맞지 않는 이슈는 유지하지 않는다.
새 기획으로 방향이 바뀌면, 기존 오픈 이슈는 닫고
이 문서 기준으로 다시 등록한다.
