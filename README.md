## 설계 방향

### 왜 라이브러리 없이 순수 CSS?

포트폴리오는 "나를 보여주는 공간"이라고 생각했어요.  
Tailwind나 MUI를 쓰면 빠르지만, 결과물이 비슷비슷해지는 경향이 있어요.  
CSS Variable로 디자인 토큰을 직접 관리하면서 **의도한 디자인을 정확하게** 표현하는 데 집중했습니다.

### 모바일 퍼스트

모든 컴포넌트는 모바일 기준으로 먼저 스타일링하고, `min-width` 미디어 쿼리로 데스크탑을 확장하는 방식으로 작성했어요.

### 순수 함수 유틸 + TDD

Nav 스크롤 스파이(`getActiveSection`), 텍스트 말줄임(`truncateText`), 배열 그룹핑(`groupBy`) 등 재사용 유틸 함수는 스펙 문서 → 테스트 → 구현 순서로 작업했습니다.  
UI 없는 로직은 브라우저 없이도 검증 가능하기 때문에 테스트 효과가 높아요.

## AI와 협업한 방법

이 프로젝트는 AI 어시스턴트(클로)와 함께 작업했어요.  
단순히 코드를 생성시키는 게 아니라, **개발자와 AI가 역할을 나눠서** 일하는 방식을 실험했습니다.

### 워크플로

```
이가은 (기획/결정)          클로 (구현/실행)
─────────────────────────────────────────
이슈 방향 확인 →            이슈 단위 브랜치 생성
                            코드 작성 + 테스트
PR 리뷰 →                  PR 생성
머지 결정 →                 다음 이슈로 진행
```

### 실제로 협의한 것들

- **브랜치 전략**: feature 브랜치를 이슈 단위로 쪼개자 (`feat/issue-{번호}-{설명}`)
- **테스트 범위**: 컴포넌트는 스킵, 순수 함수 유틸만 TDD 적용
- **스크린샷 기준**: 반응형 구현은 모바일 퍼스트, 리뷰/확인은 PC 뷰포트
- **CI 설정**: PR마다 테스트 자동 실행, 실패 시 머지 블록

이 과정에서 "AI가 만든 코드"가 아니라 **내가 이해하고 리뷰한 코드**를 머지하는 경험을 쌓았어요.

## 프로젝트 구조

```
fe-portfolio/
├── src/
│   ├── components/       # UI 컴포넌트
│   │   ├── Header/       # sticky nav + 스크롤 스파이
│   │   ├── Hero/         # 인트로 섹션
│   │   ├── Skills/       # 기술 스택 태그
│   │   ├── Projects/     # 프로젝트 카드
│   │   ├── Contact/      # 연락처
│   │   └── Footer/
│   ├── hooks/            # 커스텀 훅 (useScrollSpy 등)
│   ├── utils/            # 순수 함수 유틸 + 테스트
│   ├── data/             # 프로젝트/스킬 데이터
│   └── types/            # 공통 타입 정의
├── docs/                 # 설계 문서, 컨벤션, 스펙
└── .github/workflows/    # CI (테스트 자동화)
```

## 로컬 실행

```bash
npm install
npm run dev       # 개발 서버
npm run test      # 테스트 (watch 모드)
npm run test:run  # 테스트 1회 실행
npm run build     # 프로덕션 빌드
```
